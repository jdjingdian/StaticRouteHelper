## Context

当前安装状态在 UI 层仅体现“已安装/未安装”，而 macOS 14+ 下 `SMAppService` 的 `.requiresApproval` 实际表示“已安装但未获后台运行授权”。现状把该状态映射为未安装，导致安装成功后用户看到错误状态，并且因守卫逻辑阻断卸载，用户无法在该中间态自助恢复。

## Goals / Non-Goals

**Goals:**
- 在状态模型中引入“待生效”语义，准确表达“已安装 + 后台开关关闭”。
- 安装成功后如检测到 `.requiresApproval`，立即进入待生效状态并在 UI 以黄色感叹号提示。
- 待生效状态下允许卸载，并提供可执行的 `osascript` 卸载路径。
- 保持现有安装方式（SMAppService / SMJobBless）兼容，不影响已安装可用路径。

**Non-Goals:**
- 不重构整个安装架构或替换既有安装技术栈。
- 不改变 macOS 12–13 的能力边界（无 SMAppService 后台开关语义）。
- 不在本次变更中新增复杂诊断 UI（如多级故障排查向导）。

## Decisions

1. **新增显式状态值而非复用 notInstalled**
   - 决策：在 Helper 状态映射层新增 `pendingActivation`（命名可按现有枚举风格落地），用于承载 `.requiresApproval`。
   - 原因：`notInstalled` 表示“二进制/服务不存在”，与 `.requiresApproval` 语义冲突；显式状态可避免 UI 与操作策略分叉。
   - 备选：继续复用 `notInstalled` 并靠文案区分；放弃，因会污染卸载与横幅优先级判断。

2. **状态来源采用“安装结果 + 运行态刷新”双通道**
   - 决策：安装流程成功返回后立即刷新 `SMAppService.status`，并在后续 `refreshState()` 监听中持续维护待生效状态。
   - 原因：保证“点击安装后”即时反馈，同时覆盖用户在系统设置切换后台开关后的状态回流。
   - 备选：仅依赖周期轮询；放弃，因安装后反馈滞后。

3. **待生效状态允许卸载并走 osascript 路径**
   - 决策：当状态为待生效时，不再走“先开启后台开关”的阻断分支；改为允许卸载并调用 `osascript` 执行卸载命令。
   - 原因：待生效本质是“安装残留待授权”，必须给用户可逆操作。
   - 备选：仍要求先开启后台开关；放弃，因用户已明确该路径不可用且造成卡死感。

4. **横幅优先级将待生效置于未安装之前**
   - 决策：主窗口横幅优先显示“待生效”提示（黄色警示），其次才是“未安装”。
   - 原因：两者都可用 warning 样式，但待生效的下一步动作是“开启后台权限或卸载重装”，与未安装引导不同。
   - 备选：与未安装共用同一文案；放弃，因会误导用户重复安装。

## Risks / Trade-offs

- [风险] `osascript` 卸载在不同系统语言/权限上下文下行为不一致 → [缓解] 封装统一执行入口，保留 stderr 并写入结构化日志。
- [风险] 新状态引入后旧分支未完全覆盖，导致 UI 显示冲突 → [缓解] 以横幅优先级和状态映射单点函数集中管理，并补充场景测试。
- [取舍] 增加一个状态会提升条件分支复杂度 → [缓解] 将“状态判定”和“操作可用性”拆分为独立函数，降低耦合。

## Migration Plan

1. 扩展状态枚举与映射逻辑，确保 `.requiresApproval` 不再映射为未安装。
2. 调整安装完成后的状态刷新与 UI 绑定，确保可立即进入待生效显示。
3. 修改卸载守卫：待生效状态放行，并接入 `osascript` 卸载实现。
4. 更新横幅优先级与文案，补充本地化字符串。
5. 通过手动回归验证三条主路径：未安装、待生效、已安装。
6. 若出现异常，可回滚到旧状态映射与旧卸载守卫（不影响数据模型持久化）。

## Open Questions

- `pendingActivation` 的最终对外文案是否统一为“待生效”，还是区分“待授权/待生效”？
- `osascript` 卸载是否需要在 UI 上展示二次确认提示，以避免误触？
